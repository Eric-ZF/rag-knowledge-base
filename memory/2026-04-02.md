
## Phase 0.7 论文池管理修复（11:30）

### 根因：papers_db 和 ChromaDB 不同步
- .env 设置 CHROMADB_DIR=/tmp/chromadb，但 /root/.openclaw/rag-data/chromadb 也有旧数据
- /tmp/chromadb 是实际运行数据（191 chunks），旧目录只有 22 chunks
- 2fd0a101: papers_db 有记录，ChromaDB 无数据（phantom entry）
- 00b1336d: ChromaDB 有 22 chunks，papers_db 漏登

### 修复内容
1. **数据层**：删除 2fd0a101 phantom，补录 00b1336d
2. **去重**：pipeline.py parse_pdf() 计算 SHA256，上传时查重
3. **论文列表**：GET /papers 实时查 ChromaDB chunk 数量
4. **删除功能**：DELETE /papers/{id} 端点完善，ChromaDB + papers_db 双删
5. **前端**：论文卡片 hover 显示删除按钮，二次确认
6. **部署**：nginx 接管 8080，CORS headers 强制追加

### 关键教训
- ChromaDB 的 `count(where=...)` 行为不稳定，用 `len(get(where=...)["ids"])` 更可靠
- .env 的 CHROMADB_DIR=/tmp/chromadb 被 config.py 读取，导致数据存在 /tmp（重启丢）
- ChromaDB 的 `/tmp/chromadb` 和 `/root/.openclaw/rag-data/chromadb` 是两个完全不同的目录

## 教训：登录"没反应"排查（15:00）

### 根因
- 前端 API URL 从 `:8000` 改成 `:` (默认走 port 80)
- nginx 只监听了 8080，port 80 没人监听
- 浏览器 POST → `http://124.156.204.163/auth/login` → 连接超时

### 修复
1. 前端 API URL 改为 `http://124.156.204.163:8080`（显式端口）
2. nginx 同时监听 80 + 8080

### 模式识别：「登录没反应」= 3 种可能
1. JS 语法错误（浏览器静默）→ 浏览器 Console 直接看
2. `r.json()` 调用两次吞掉错误 → 最难定位，今天排了1小时
3. 连接超时 → 看 Network 面板 `ERR_CONNECTION_TIMEDOUT`
