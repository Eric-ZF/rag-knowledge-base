## 1. 产品定位与愿景

### 一句话定位

> **"会理解你论文的智能研究伙伴"**
> 不是搜索引擎，不是笔记工具，是能读懂你论文并回答你问题的研究助手。

### 核心洞察

RAG 知识库不是搜索引擎的替代，是**研究工作流的重构**：

| 工具 | 在做什么 | 我们的差异 |
|-----|---------|---------|
| Kimi / 秘塔 | 公网 AI 搜索，答完即忘 | 答案来自**你自己的论文库**，有私有性 |
| Elicit / Consensus | 英文为主，无中文深度 | **中国核心期刊**是真实的市场空白 |
| Zotero / Mendeley | 论文管理 + 引用格式化 | 我们不只是管理，是**理解 + 问答** |
| Notion / Obsidian | 通用笔记，无 PDF 语义理解 | 我们的索引是**论文全文语义**，不是通用笔记 |

### 核心壁垒演进路径

```
短期（<6个月）: 中国核心期刊 PDF 深度覆盖 × 中文 RAG 效果
         ↓
长期（>12个月）: 用户私有知识资产积累 → 形成个人研究知识网络
         ↓
护城河形成: 知识网络越大，新论文推荐准确性越高 → 迁移成本出现
```

## 3. MVP 定义

### 核心原则：只押注一个假设

> **我们验证的核心假设只有一个：研究者愿意为"针对自己论文库的智能问答"付钱。**

一切工程资源优先验证这一个假设，其余全部推迟。

### MVP 功能范围

| 功能 | MVP | Phase 2+ | 说明 |
|-----|-----|---------|------|
| PDF 上传 | ✅ | — | 只支持 PDF，不做其他格式 |
| 论文列表 | ✅ | — | 基础列表（时间/标题排序），无完整文件夹体系 |
| **RAG 问答** | ✅ | — | **核心差异化功能** |
| 答案引用标注 | ✅ | — | 点击跳转到原文对应位置 |
| 语义检索 | ✅ | — | 基础 ANN 检索 |
| 收藏夹（完整体系）| ❌ | ✅ | Phase 2：文件夹嵌套/标签/多级分类 |
| 批量选择操作 | ❌ | ✅ | Phase 2：MVP 只做单篇操作 |
| **团队协作/共享** | ❌ | ❌ | **不做（CEO v2.0 明确砍掉）** |
| 知识图谱可视化 | ❌ | ✅ | Phase 2：MVP 只做向量检索 |
| 写作插件 | ❌ | ❌ | **不做** |

## 4. 两阶段验证与发布计划

### Phase 0：验证 Sprint（第 1-2 个月）

> **目标：用手动的、半自动化的方式，快速验证用户是否真的愿意付钱。**

| 周次 | 动作 | 验证指标 |
|-----|------|---------|
| 第 1 周 | 选定切入点：研究生"文献综述"场景 | 确定一个足够痛的具体场景 |
| 第 2-3 周 | 用 GPT-4 + LangChain 手动跑通 pipeline，不做产品化 | 技术可行性验证 |
| 第 4 周 | 邀请 20 个真实用户内测（免费） | 观察：是否主动推荐给同学 |
| 第 5-8 周 | 开放公测，收集付费意向（不用完整产品）| "如果现在收费，你愿意付多少？" |

**通过标准（必须同时满足）：**
- ✅ 内测用户每周使用 RAG 问答 ≥ 5 次
- ✅ 超过 50% 的用户主动问"什么时候可以付费"
- ✅ 没有人因为"担心隐私"拒绝上传真实论文

**如果未通过：立即 Pivot 或放弃**

### Month 3-5：MVP 产品化

| 月份 | 目标 | 关键交付 |
|-----|------|---------|
| M3 | 基础架构上线 | Next.js + FastAPI + Qdrant 上线，50 种子用户迁移 |
| M4 | 核心功能完善 | RAG 质量达标（引用召回 ≥ **60%**，Phase 2 目标 ≥ 85%），搜索延迟 P99 < 2s |
| M5 | 付费层上线 | Free/Pro/Academic 三档定价，Stripe 集成 |

### Month 6-12：增长 + Phase 2 探索

**Phase 2 探索功能（暂定，不承诺）：**
- 收藏夹 + 标签管理体系
- 中文核心期刊元数据深度覆盖
- 论文推荐（基于已有论文的相似论文发现）

**不做（明确砍掉）：**
- ❌ 团队协作/共享收藏夹
- ❌ 写作插件集成

| 前端 | Next.js (SPA) | 桌面端 BS 架构，研究者在各设备使用 |

### 8.3 核心数据流

**论文上传 → 索引 pipeline：**

```
用户上传 PDF
    │
    ▼
S3 (AES-256 加密存储，路径含 paper_id，不含 user_id)
    │
    ▼
PDF Worker (内存处理，不落盘)
    │
    ▼
Unstructured.io → Markdown 文本（保留公式/表格）
    │
    ▼
分块 (chunk_size=512 tokens)
    │
    ▼
Embedding API (text-embedding-3-small)
    │
    ▼
Qdrant (user_id filter 强制隔离)
    │
    ▼
S3 原文件标记删除（24h 内物理删除）
```

**RAG 问答 pipeline：**

```
用户问题
    │
    ▼
Embedding → 向量
    │
    ▼
Qdrant 检索 (top_k=5, user_id filter)
    │
    ▼
上下文组装 (chunks + paper_meta)
    │
    ▼
Claude 3.5 Haiku → 带引用的答案
    │
    ▼
流式返回 (SSE)
```

### 8.4 数据库设计

**PostgreSQL Schema（核心表）：**

```sql
-- 用户表
CREATE TABLE users (
  id UUID PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255),
  plan VARCHAR(20) DEFAULT 'free',  -- free / pro / academic
  storage_used_bytes BIGINT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 论文元数据
CREATE TABLE papers (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  authors TEXT[],
  year INTEGER,
  venue TEXT,
  doi VARCHAR(255),
  file_path VARCHAR(512),    -- S3 加密路径
  file_hash VARCHAR(64),     -- SHA256，防重复上传
  status VARCHAR(20) DEFAULT 'pending',  -- pending/indexing/ready/error
  chunk_count INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 向量索引任务
CREATE TABLE indexing_jobs (
  id UUID PRIMARY KEY,
  paper_id UUID REFERENCES papers(id),
  status VARCHAR(20),   -- queued/running/done/failed
  chunks_indexed INTEGER DEFAULT 0,
  error_msg TEXT,
  started_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ
);
```

**Qdrant Collection：**

```json
{
  "name": "paper_chunks",
```

## 9. 前端交互设计

### 9.1 设计理念

> **"安静的学术感"** — 像翻开一本精装书、研究者工作台那样沉稳、精致、有温度。

| 关键词 | 实现 |
|-------|------|
| **Material** | 纸张纹理、微阴影层次、磨砂玻璃（仅固定元素）、边框光泽 |
| **Quiet** | 克制配色、留白充足、动效轻柔（220ms 为主）|
| **Precise** | 4px 基础单位网格、精确间距层级 |

## 11. 安全评估

### 11.1 五大安全威胁（S1-S5 为 MVP 前必修复项）

| ID | 威胁 | 组件 | 风险 | 修复方案 |
|----|------|------|------|---------|
| **S1** | JWT 算法弱（HS256/none）| API | 🔴 严重 | 必须使用 RS256/ES256 |
| **S2** | S3 PDF URL 无签名验证 | S3 | 🔴 严重 | 生成签名 URL（5min 有效期）+ user_id 所有权验证 |
| **S3** | Qdrant 无认证，跨用户向量泄露 | 向量库 | 🔴 严重 | 每用户独立 collection + API Key 认证 |
| **S4** | SQL 注入 | PostgreSQL | 🔴 严重 | ORM + 参数化查询，永不拼接 |
| **S5** | PDF 文件名路径遍历 | Worker | 🔴 高 | UUID 路径，不使用用户文件名 |
| **S6** | 暴力破解登录 | Auth | 🔴 高 | 5 次/15min 限流 + Rotation |
| **S7** | Prompt 注入（PDF 内容逃逸）| RAG | 🟠 中 | 用户输入与 system prompt 严格隔离 |

### 11.2 零信任架构

```
Never Trust, Always Verify.
每一次访问都必须验证。
```

**实施清单：**

| 领域 | 措施 |
|-----|------|
| **身份认证** | JWT (RS256) + 1h TTL；Refresh Token Rotation（7d，一次性）；登录限流（5次/15min）|
| **授权** | 最小权限；PostgreSQL RLS（行级安全）；Qdrant collection 隔离；S3 签名 URL |
| **网络** | 全站 HTTPS (TLS 1.3)；HSTS；CORS 白名单（非 *）；内部服务不暴露公网 |
| **数据** | 静态加密（AES-256-KMS）；传输加密（TLS）；无明文密码（bcrypt）；PDF 原文件 24h 内删除 |
| **监控** | 结构化安全日志（JSON）；登录失败告警；配额超限告警 |

### 11.3 隐私合规

| 原则 | 实施 |
|-----|------|
| **目的限制** | 用户数据只用于 RAG 问答，不用于模型训练，不给第三方 |
| **最小化** | 只存向量和元数据，原始 PDF 处理后即删 |
| **存储限制** | 用户删除论文后，向量和元数据立即清除 |
| **被遗忘权** | 用户注销后 30 天内删除全部个人数据 |

## 12. 关键工程决策（ADR）

### ADR #1：整体风格
**决策**：Phase 0 纯手动验证 → Month 3-5 MVP 单体 → Month 6+ 按需拆分
**理由**：Kent Beck 原则，"先跑起来再演进"

### ADR #2：隐私路线
**决策**：PDF 在**服务端内存中完成解析+向量化**，原文件处理完立即删除，只存储向量+元数据
**理由**：功能完整 vs 隐私的平衡，参考 S2/S3 安全要求

### ADR #3：Embedding 选型
**决策**：OpenAI text-embedding-3-small（Phase 1-2）→ BGE-large-zh 自托管（Phase 3）
**理由**：Phase 1 零运维验证产品，Phase 3 中文能力 + 降本

### ADR #4：向量数据库
**决策**：Qdrant（开源，Rust，HNSW + 混合检索）
**理由**：比 Milvus 轻量，比 Pinecone 便宜，支持本地部署

### ADR #5：LLM 选型
**决策**：Claude 3.5 Haiku（主力） + GPT-4o-mini（备用）
**理由**：速度快，学术内容理解强，成本低

### ADR #6：PDF 预览
**决策**：浏览器端 `pdf.js` 渲染，不做服务端预览 API
**理由**：避免 PDF 预览 API 成为数据泄露点（S2 威胁）

### ADR #7：三档定价配额
**决策**：Free 20 篇 / Pro 无限 / Academic 无限（5人团队）
**理由**：与 CEO v2.0 定价方案对齐，PostgreSQL + Redis 实现
